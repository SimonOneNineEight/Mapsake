# Decision Log — travel-map PRD

Canonical memory and audit trail for the PRD. Every decision, change, and override recorded here.

## 2026-06-16

- Intent: **Create**. No prior PRD. Foundation inputs: product brief (`briefs/brief-travel-map-2026-06-16/brief.md`, status final), brief addendum, brief decision log, brainstorming session, and saved concept memory.
- Stake read (inherited from brief): **passion project / personal-tool-first.** Primary user is Simon; v1 success = he returns to re-live trips. Right-size rigor accordingly.
- No new inputs since the brief; brief stands as the complete foundation.
- Working mode: **Coaching path** chosen by Simon. Walk PM-thinking sections together before drafting.
- Entry point: **Journey-led.** Section order driven by user journeys; features derived from them. Rationale: consumer/UX-heavy product whose reason to exist is an emotional loop.
- **Notification model (Simon's call):** follow Apple Photos "on this day" pattern — short, specific text built from place + date (e.g. "2 years ago today: Kyoto"). Proven, cheap to build, matches Simon's own behavior. Reinforces brief principle: memory delivery, never engagement nag.
- **Re-live landing (Simon's call):** tapping the notification lands on **map + memory together** — map flies/zooms to the place, place is highlighted, and the memory panel (photos + notes) is open simultaneously. Chosen over map-only and photo-view-first. Rationale: shows where AND what at once; keeps the map central (the product's differentiator vs. a photo app).
- **Onboarding order (Simon's call):** choose **default view first**, then backfill adapts to it. World view → tap countries on the globe. Focus-on-a-country view → drop into that country and tap its admin-1 regions. Rationale: meets the user where their travel actually is (e.g. someone who only road-trips their own country fills granular detail, not one country blob).
- **Granularity (Simon's call, locked):** v1 = **country + admin-1 sub-region**, where admin-1 is the standard first-level subdivision per country in the boundary dataset. This is data-driven and locally correct: US→states, Taiwan→counties/special municipalities (effectively cities), Japan→prefectures. Resolves the "states don't fit Taiwan" problem without a global rule. **Parked as fast-follow:** dedicated city-pins *within* large admin-1 regions (e.g. pinning Los Angeles inside California) — needs a separate points/pins data model and carries the real cost.
- **Focus-country view model (Simon confirmed):** "focus on a country" view simply asks *which* country at onboarding; that country is the landing view and backfill starts at its admin-1 regions. No separate "home country" concept to manage — it's just the chosen focus country. Changeable in settings.
- **All three user journeys confirmed by Simon:** UJ-1 (first fill-in: choose view → backfill adapts), UJ-2 (re-live loop), UJ-3 (log a fresh trip). These drive the feature set.
- **Feature spine derived from journeys** (FR1–FR20), grouped A Accounts & Durability / B Map / C Place Memories / D Onboarding / E Retention loop. Presented to Simon.
- Full PRD draft assembled (prd.md) + addendum.md written. During assembly the PWA install prompt was placed in the Onboarding group as **FR17** (it's load-bearing for the loop), shifting the retention FRs to FR18–FR21. Success metrics/counter-metrics and scope carried forward from the brief; NFR1–6 derived; risks/open items captured (PWA dependency, map-data geopolitics, auth, photo storage, notification cadence).
- **Product name (Simon's call): Mapsake.** Chosen after a naming exploration. "Mapsake" = map + keepsake ("-sake" from "for the sake of"); reads as "the map you keep, for the sake of memory" — lands on the private/treasured/durable positioning.
  - Availability check (web/app search proxy, not a registrar/trademark check): no app or real web presence found for "Mapsake" — appears clear.
  - Rejected after availability checks: Wander (taken — wander.com luxury homes + social net + Wanderlog), Revisit (taken — travel photo-memory game + travel spot organizer), Wayfare (crowded — multiple travel apps + agencies), Mapory (taken — near-identical concept: photos/notes on a personal map), Sojourn (taken — Your Sojourn social travel app), Everbeen (derivative of the "Been" cluster), Trovel (close neighbor "Trovelo" trip planner), Travmem (Simon's idea; mostly free but reads abbreviation-y + nearby "TravMemo" brand).
  - **[OPEN] Pending verification before launch:** domain registration (e.g. mapsake.com) and USPTO trademark search. Recorded as an open item.
  - "travel-map" remains the workspace/codename; PRD title and overview updated to Mapsake.
- **Notification delivery (Simon's call):** **web push + PWA install prompt.** Onboarding nudges users to add the app to home screen so web push works (incl. iPhone, which requires PWA install). On-brand (stays in-app). **Tracked risk:** the retention loop's reach on iPhone depends on PWA install; if users skip install, the core "on this day" buzz never fires on their phone. Mitigation: load-bearing onboarding install nudge; monitor install + notification-open rates post-launch. Email-as-carrier considered and not chosen for v1.
